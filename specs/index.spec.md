# Specs

Specs describe the **desired state** of Lockwire. Code is the actual state. Development work reconciles the two.

## Principles

1. **Desired state, not issue tracking.** A spec says "the system SHALL do X" — not "here's a bug." If the system doesn't match the spec, the code is wrong.
2. **Living documents.** Specs are amended or deleted, never archived.
3. **Behavior contracts, not implementation plans.** Specs describe observable behavior — inputs, outputs, error conditions, constraints. Library choices and internal structure belong elsewhere.
4. **Organized by capability domain.** Add domains when existing ones become too broad, not preemptively.

## Domain Vocabulary

| Term | Definition |
|------|-----------|
| **Session** | An ephemeral terminal sharing instance. Has a Code, a Stream Key, and zero or more Viewers. Exists only while the Sharer's process runs. |
| **Code** | A six-word phrase generated at session creation. Used locally as the SPAKE2 password and as the Argon2id input for Session ID derivation. Never transmitted, never used to derive the Stream Key. Shared by the user out-of-band. |
| **Sharer** | The operator who runs `lw share` and controls the shared terminal. |
| **Viewer** | A participant who joined via `lw join <code>`. Read-only access. |
| **Stream Key (K)** | A 256-bit random symmetric key generated by the Sharer. Never shown to the user. Never derived from the Code. |
| **Session ID** | A relay-facing routing token derived from the Code via Argon2id. Both Sharer and Viewer compute it independently before connecting to the relay. The relay uses it to route messages without knowing K or Code. |
| **Epoch** | A 60-second time window, identified by a monotonically increasing integer (`floor(unix_time / 60)`). |
| **Epoch Key (K_n)** | A subkey derived from K for epoch n. Used for stream encryption within that epoch. |
| **Relay** | A server that forwards encrypted blobs between Sharer and Viewers. Knows only the Session ID. Cannot read content. |

## Format

```markdown
# <Domain> Specification

## Purpose
High-level description of this spec's domain.

## Requirements

### Requirement: <Name>
The system SHALL <observable behavior>.

#### Scenario: <Name>
- GIVEN <precondition>
- WHEN <action>
- THEN <expected outcome>
- AND <additional outcome>
```

## What belongs in a spec

- Observable behavior users or downstream systems rely on
- Inputs, outputs, and error conditions
- External constraints (security, privacy, reliability, compatibility)
- Scenarios that can be tested or explicitly validated

## What does not belong in a spec

- Internal class/function names
- Library or framework choices
- Step-by-step implementation details

**Quick test:** if the implementation can change without changing externally visible behavior, it does not belong in the spec.

## Directory Structure

```
specs/
  index.spec.md          # This file — format, vocabulary, structure
  session/               # Session lifecycle and viewer management
  encryption/            # Cryptographic model
  relay/                 # Relay server behavior and constraints
  cli/                   # CLI commands and UX
  web-client/            # Browser-based viewer
```
